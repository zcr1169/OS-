## 三、调度器（核心演示）

> 本节是验收重点。构建 CPU 密集型进程，覆盖三个队列，直观展示升降级和交替运行。
> 
> 从第二节结束时的状态继续（进程已全部清空，仅剩 init(PID=0)）。若当前有残留进程，先用 `kill_pcb` 清理。

### 3.1 构建三个 CPU 密集型测试进程

```bash
create_pcb worker_A 2 1 10
create_pcb worker_B 5 1 10
create_pcb worker_C 10 1 10
```

三个进程 burstTime=10（需要 10 个 CPU 时间才能跑完）。

> **标准 MLFQ：新进程一律进 Q0**，不论 priority 参数。priority 仅记录在 PCB 中，不影响入队位置。三个进程创建后全部在 Q0，等待轮转调度。

### 3.2 自动调度 —— 观察完整降级链

```bash
start_sched
```

每 2秒 自动输出一行调度日志。约 15~20 秒跑完，关键输出：

```text
调度器已启动 (多级反馈队列)
Q0(优先级 0-3): worker_A(1) -> worker_B(2) -> worker_C(3)
Q1(优先级 4-7): (空)
Q2(优先级 8-15): (空)
```

> 三个进程全部在 Q0，按 FCFS 排列。

**第一轮：Q0 时间片=2，逐个降级到 Q1**

```text
[调度] >> 选中 PID=1 (worker_A) 来自 Q0(优先级 0-3) | 时间片=2 | CPU=2/10
   决策: Q0 非空 → 取 worker_A → 时间片 2 | CPU 2 < 10 → 时间片耗尽, 降级 Q0→Q1
[调度] >> 选中 PID=2 (worker_B) 来自 Q0(优先级 0-3) | 时间片=2 | CPU=2/10
   决策: Q0 非空 → 取 worker_B → 时间片 2 | CPU 2 < 10 → 时间片耗尽, 降级 Q0→Q1
[调度] >> 选中 PID=3 (worker_C) 来自 Q0(优先级 0-3) | 时间片=2 | CPU=2/10
   决策: Q0 非空 → 取 worker_C → 时间片 2 | CPU 2 < 10 → 时间片耗尽, 降级 Q0→Q1
```

> Q0 三个进程依次执行，各自用完时间片 2 后全部降到 Q1。

**第二轮：Q0 空 → 调度 Q1，时间片=4，逐个降级到 Q2**

```text
[调度] >> 选中 PID=1 (worker_A) 来自 Q1(优先级 4-7) | 时间片=4 | CPU=6/10
   决策: Q0 空, Q1 非空 → 取 worker_A → 时间片 4 | CPU 6 < 10 → 时间片耗尽, 降级 Q1→Q2
[调度] >> 选中 PID=2 (worker_B) 来自 Q1(优先级 4-7) | 时间片=4 | CPU=6/10
   决策: Q1 非空 → 取 worker_B → 时间片 4 | CPU 6 < 10 → 时间片耗尽, 降级 Q1→Q2
[调度] >> 选中 PID=3 (worker_C) 来自 Q1(优先级 4-7) | 时间片=4 | CPU=6/10
   决策: Q1 非空 → 取 worker_C → 时间片 4 | CPU 6 < 10 → 时间片耗尽, 降级 Q1→Q2
```

> Q0、Q1 都空后才轮到 Q2。三个进程各自累加到 CPU=6，全部降到 Q2。

**第三轮：Q0、Q1 空 → 调度 Q2，时间片=8，逐个完成**

```text
[调度] >> 选中 PID=1 (worker_A) 来自 Q2(优先级 8-15) | 时间片=8 | CPU=14/10
   决策: Q2 非空 → 取 worker_A → 时间片 8 | CPU 14 >= 10 → 进程完成! 已自动终止
[调度] >> 选中 PID=2 (worker_B) 来自 Q2(优先级 8-15) | 时间片=8 | CPU=14/10
   决策: Q2 非空 → 取 worker_B → 时间片 8 | CPU 14 >= 10 → 进程完成! 已自动终止
[调度] >> 选中 PID=3 (worker_C) 来自 Q2(优先级 8-15) | 时间片=8 | CPU=14/10
   决策: Q2 非空 → 取 worker_C → 时间片 8 | CPU 14 >= 10 → 进程完成! 已自动终止

[调度] 所有用户进程已完成, 调度器自动停止
```

> Q2 时间片=8 足够三个进程一次性跑完（只需再 4 个时间单位）。

**验收要点**：

- 三个进程全部从 Q0 出发，经历 **Q0→Q1→Q2** 完整三级降级
- 降级是纯粹逐级降：Q0 耗尽 → Q1，Q1 耗尽 → Q2，**无任何跳级**
- 每轮内三个进程按 FCFS 顺序轮转，高优先级队列全部为空后才调度低优先级队列
- CPU 数值在动态增长，不是直接改状态

### 3.3 单步演示 —— 完整决策链路

> 核心要求：每一步必须展示「队列当前状态 → 选中谁 → 决策逻辑 → 执行结果」的完整推导。

清空旧进程，创建新测试进程。burstTime=6 确保多步才能跑完：

```bash
kill_pcb 1
kill_pcb 2
kill_pcb 3
create_pcb s_a 3 1 6
create_pcb s_b 8 1 6
```

s_a 和 s_b 都进 Q0（标准 MLFQ：新进程一律进 Q0）。

#### 第一步

```bash
step
```

```text
当前队列状态:
  Q0(优先级 0-3): s_a(1)[0/6] -> s_b(2)[0/6]
  Q1(优先级 4-7): (空)
  Q2(优先级 8-15): (空)
[调度] >> 选中 PID=1 (s_a) 来自 Q0(优先级 0-3) | 时间片=2 | CPU=2/6
   决策: Q0 非空 → 取 s_a → 时间片 2 | CPU 2 < 6 → 时间片耗尽, 降级 Q0→Q1
   执行后队列: Q0=s_b(2)[0/6] | Q1=s_a(1)[2/6]
```

> **决策推导**：扫描 Q0=[s_a, s_b], Q1=[], Q2=[]。Q0 非空 → 取队首 s_a → 时间片 2 → CPU 2 < 6 未完成 → 降级到 Q1。现在 Q0=[s_b], Q1=[s_a]。

#### 第二步

```bash
step
```

```text
当前队列状态:
  Q0(优先级 0-3): s_b(2)[0/6]
  Q1(优先级 4-7): s_a(1)[2/6]
  Q2(优先级 8-15): (空)
[调度] >> 选中 PID=2 (s_b) 来自 Q0(优先级 0-3) | 时间片=2 | CPU=2/6
   决策: Q0 非空 → 取 s_b → 时间片 2 | CPU 2 < 6 → 时间片耗尽, 降级 Q0→Q1
   执行后队列: Q1=s_a(1)[2/6] -> s_b(2)[2/6]
```

> **决策推导**：Q0=[s_b], Q1=[s_a], Q2=[]。Q0 非空 → 取 s_b → 时间片 2 → CPU 2 < 6 → 降级到 Q1。现在 Q1=[s_a, s_b]，两个进程都在 Q1。

#### 第三步

```bash
step
```

```text
当前队列状态:
  Q0(优先级 0-3): (空)
  Q1(优先级 4-7): s_a(1)[2/6] -> s_b(2)[2/6]
  Q2(优先级 8-15): (空)
[调度] >> 选中 PID=1 (s_a) 来自 Q1(优先级 4-7) | 时间片=4 | CPU=6/6
   决策: Q0 空, Q1 非空 → 取 s_a → 时间片 4 | CPU 6 >= 6 → 进程完成! 已自动终止
   执行后队列: Q1=s_b(2)[2/6]
```

> **决策推导**：Q0=[], Q1=[s_a, s_b], Q2=[]。Q0 空 → 调度 Q1 → 取 s_a → 时间片 4 → CPU 6 ≥ 6 → 完成！s_a 终止。队列剩下 Q1=[s_b]。

#### 第四步

```bash
step
```

```text
当前队列状态:
  Q0(优先级 0-3): (空)
  Q1(优先级 4-7): s_b(2)[2/6]
  Q2(优先级 8-15): (空)
[调度] >> 选中 PID=2 (s_b) 来自 Q1(优先级 4-7) | 时间片=4 | CPU=6/6
   决策: Q0 空, Q1 非空 → 取 s_b → 时间片 4 | CPU 6 >= 6 → 进程完成! 已自动终止
   执行后队列: (全空)
```

> s_b 在 Q1 时间片 4 内直接跑完（只需再 4），四步调度演示完毕。

#### 第五步

```bash
step
```

```text
所有调度队列为空, 无可调度进程
```

#### 第六步

```bash
step
```

```text
所有调度队列为空, 无可调度进程
```

### 4.2 交错分配 —— 制造碎片潜力

> 关键：交替分配给进程和 data，产生交错布局，为后续释放制造碎片做准备。

```bash
create_pcb proc_a 3 1 10
create_pcb proc_b 5 1 10
alloc 100 1
alloc 200 data
alloc 100 2
alloc 100 1
show_mem
```

预期内存分配图：

```text
|##PID1(100KB)|##数据(200KB)|##PID2(100KB)|##PID1(200KB)|--free(424KB)--|
```

> 四个已分配块交错排列，空闲块在末尾。

### 4.3 释放中间块 —— 观察 auto-compact

> `free_mem` 内部自动调用 `compactInternal()`，相邻空闲块会立即合并。

```bash
free_mem 100
show_mem
```

释放数据块（addr=100, 200KB），预期内存分配图：

```text
|##PID1(100KB)|--free(200KB)--|##PID2(100KB)|##PID1(200KB)|--free(424KB)--|
```

> 产生了一个 200KB 的空洞，但与末尾 424KB 不相邻（PID3 和 PID2 块隔在中间），无法合并。

再释放 PID3（addr=300, 100KB）：

```bash
free_mem 300
show_mem
```

预期内存分配图：

```text
|##PID1(100KB)|--free(300KB)--|##PID1(200KB)|--free(424KB)--|
```

> PID3 的 100KB 释放后与前面 200KB 空闲块**相邻**，auto-compact 自动合并为 300KB。

### 4.4 再释放一个 —— 观察连续合并

```bash
free_mem 400
show_mem
```

释放 PID2 的 200KB（addr=400），预期内存分配图：

```text
|##PID1(100KB)|--free(924KB)--|
```

> 200KB 释放后与前面 300KB、后面 424KB **全部相邻**，三块一次合并为 924KB。
> 这就是 auto-compact 的核心价值：每次释放都自动合并相邻空闲块，防止碎片累积。

### 4.5 内存统计

```bash
mem_stat
```

预期输出：

```text
===== 内存统计 =====
总内存:     1024 KB
已分配:     100 KB
空闲:       924 KB
空闲块数:   1
碎片率:     0.0 %
分配算法:   首次适应(FF)
```

> 空闲块数=1、碎片率=0%，说明 auto-compact 已经把所有空闲合并为一整块。

### 4.6 切换分配算法

```bash
set_alloc_algo BF
set_alloc_algo WF
set_alloc_algo FF
```

> BF=最佳适应（选最小够用的空闲块）、WF=最坏适应（选最大的空闲块）、FF=首次适应（默认）。
> 切换后下次 `alloc` 会使用新算法。

## 七、多实例并发共享

> 重点：不同用户只能查看自己的进程（list_pcb / ptree），但 overview 可查看全局内存和调度状态。
> 通过两个不同用户验证用户隔离和全局数据共享。

### 环境准备

打开两个 PowerShell/CMD 窗口，都进入 `生成/` 目录：

```bash
# 窗口1
cd D:\AAA\OS\keshe\生成
.\OSCD.exe

# 窗口2
cd D:\AAA\OS\keshe\生成
.\OSCD.exe
```

窗口1 先启动（抢到文件锁，成为后端），窗口2 后启动（成为观察者）。

```text
窗口1:  OS Simulator v1.0 [后端]   |  help 帮助  exit 退出
窗口2:  OS Simulator v1.0 [观察者]  |  help 帮助  exit 退出
```

两个窗口分别注册并登录**不同用户**：

**窗口1（后端）— 用户 alice：**

```bash
register alice 123456
login alice 123456
```

**窗口2（观察者）— 用户 bob：**

```bash
register bob 654321
login bob 654321
```

> 观察者端注册/登录命令通过 `commands.txt` 发送到后端执行。

---

### 验证1：用户隔离 — 进程管理命令

**窗口1（alice）** 创建进程：

```bash

```

```text

```

**窗口2（bob）** 等待 1-2 秒后查看：

```bash

```

```text
PID   名称          状态    优先级  内存(KB)  所属用户      子进程
------------------------------------------------------------------
(无用户进程，仅剩 system 进程 init 不显示)
```

> **证明**：bob 的 list_pcb 看不到 alice 的进程。进程管理命令按 owner 隔离。

**窗口2（bob）** 用 ptree 验证：

```bash

```

```text

```

> bob 的 ptree 也看不到 alice 的进程树。

**窗口2（bob）** 用 show_pcb 验证（show_pcb 按 PID 查，不过滤 owner）：


```text

PID:       1
名称:      proc_a1
...
所属用户:  alice
```

> **注意**：show_pcb 不过滤 owner，任何用户都可以查看任意 PID 的详细信息。这是设计如此（全局可见），不是 bug。

### 验证6：后端断开 → 观察者自动升级

关掉窗口1（alice / 后端），窗口2（bob / 观察者）的轮询线程会自动尝试获取文件锁。当获锁成功时：

## 八、多线程日志测试

### 8.1 开启线程日志

```bash
log_on
```

预期输出：

```text
线程日志已开启
```

此后每个命令的输入输出都会在前台线程和后台线程之间穿插日志，格式为 `[前台][线程ID] 收到命令: xxx → 推入消息队列`、`[后台][线程ID] 取出命令: xxx → 开始执行`、`[后台][线程ID] 命令执行完毕, 结果已推入 resultQueue`。

### 8.2 日志模式下执行命令

```bash
create_pcb log_test 5 1 5
```

预期输出会在正常结果前出现类似以下内容：

```text
[前台][23320] 收到命令: create_pcb log_test 5 1 5 → 推入消息队列
[前台][23320] 等待后台执行结果...
[后台][14304] 取出命令: create_pcb log_test 5 1 5 → 开始执行
[后台][14304] 命令执行完毕, 结果已推入 resultQueue
进程创建成功! PID=...
```

> 日志清晰展示了前台线程接收命令→推入消息队列→后台线程取出执行→返回结果的全过程，体现了前后台线程分离的架构。

### 8.3 关闭线程日志

```bash
log_off
```

预期输出：

```text
线程日志已关闭
```

此后恢复正常输出，不再显示线程日志前缀。
